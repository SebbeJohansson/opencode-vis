## Context

The project (`@xenodrive/vis`) currently uses pnpm@10.29.3 as its package manager, enforced via the `packageManager` field in `package.json` and Corepack. The repo has a `pnpm-lock.yaml` lockfile and a `pnpm-workspace.yaml` (though no workspace package globs are defined — it's a single-package repo). Both CI workflows use `pnpm/action-setup@v4` and `pnpm -r install --frozen-lockfile`.

The goal is to fully migrate to yarn, replacing every pnpm-specific artifact.

## Goals / Non-Goals

**Goals:**
- Replace `pnpm-lock.yaml` with a `yarn.lock` generated by yarn
- Update `packageManager` field in `package.json` to yarn
- Remove `pnpm-workspace.yaml` (no workspace packages are defined)
- Update both CI workflows to use yarn for install/build steps
- Ensure `yarn install` and `yarn run build` work correctly

**Non-Goals:**
- Changing any dependencies or dependency versions
- Migrating to yarn workspaces or a monorepo structure
- Changing build tooling (Vite, etc.)

## Decisions

### Use Yarn v1 (Classic) vs Yarn v4 (Berry)

**Decision**: Use Yarn v4 (Berry) with `nodeLinker: node-modules` for compatibility.

**Rationale**: Yarn v4 is the current stable release and is what `yarn` installs by default via Corepack today. Using `nodeLinker: node-modules` avoids Plug'n'Play compatibility issues with tools like Vite that expect a standard `node_modules` layout.

**Alternatives considered**:
- Yarn v1 Classic: Still widely used but unmaintained. Berry is the supported path forward.
- PnP mode (Yarn Berry default): Would require `.pnp.cjs` and editor SDK setup — unnecessary complexity for a simple migration.

### CI Workflow Strategy

**Decision**: Replace `pnpm/action-setup@v4` with Corepack enablement (`corepack enable`) and rely on the `packageManager` field in `package.json` to drive the correct yarn version.

**Rationale**: `actions/setup-node@v4` supports `cache: yarn` natively. Corepack reads `packageManager` from `package.json`, so no separate yarn-install action is needed. This keeps CI config minimal.

**Alternatives considered**:
- `borales/actions-yarn`: Third-party action; unnecessary when Corepack handles version pinning.

### Remove pnpm-workspace.yaml

**Decision**: Delete `pnpm-workspace.yaml` entirely.

**Rationale**: The file only contains `onlyBuiltDependencies: [esbuild]` — no workspace package globs. This pnpm-specific lifecycle hook has no yarn equivalent needed; esbuild's postinstall runs automatically under yarn.

## Risks / Trade-offs

- **[Risk] Lockfile resolution differences** → Yarn may resolve transitive dependency versions differently than pnpm. Run `yarn install` and validate the build (`yarn run build`) to confirm no breakage before committing the lockfile.
- **[Risk] CI cache invalidation** → Switching from `cache: pnpm` to `cache: yarn` in `actions/setup-node` will invalidate the existing CI cache on first run. This is expected and one-time.
- **[Risk] Corepack not enabled on CI runner** → Mitigated by adding `corepack enable` as an explicit step before install in both workflows.
- **[Risk] esbuild postinstall** → pnpm's `onlyBuiltDependencies` filtered which packages ran postinstall scripts. Yarn runs all lifecycle scripts by default; esbuild's postinstall is safe and expected.

## Migration Plan

1. Update `packageManager` in `package.json` to `yarn@<version>`
2. Add `.yarnrc.yml` with `nodeLinker: node-modules`
3. Delete `pnpm-lock.yaml` and `pnpm-workspace.yaml`
4. Run `yarn install` locally to generate `yarn.lock`
5. Run `yarn run build` to verify the build succeeds
6. Update `.github/workflows/deploy.yaml`: replace pnpm setup/install/build with yarn equivalents
7. Update `.github/workflows/publish.yaml`: replace pnpm setup/install with yarn equivalents
8. Commit all changes together (lockfile + config + workflows)

**Rollback**: Revert the commit. pnpm lockfile is recoverable from git history.
